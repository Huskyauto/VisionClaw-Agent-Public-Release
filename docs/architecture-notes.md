- **Owner-jury metered daily ceiling — concurrency overshoot CLOSED via process-local reserve-then-settle (2026-06-29).** Originally the ceiling pre-check in `server/moa.ts` (`ownerJuryCeiling().exceeded`) was read-only and accrual (`noteOwnerJurySpend()`) ran AFTER the calls, so two OVERLAPPING owner-initiated jury runs in one process could both pass `exceeded` before either accrued → modest overshoot. ## Self-improvement loops — coupled-verifier Goodhart caution (R125+13.23)

**Lesson (SIA, Hexo Labs, arXiv:2605.27276):** any self-improvement loop that optimizes a *fixed* verifier risks a coupled co-evolutionary **Goodhart** failure — a fixed point that looks strong on the verifier yet stays fragile under perturbation. SIA couples two levers (harness rewrite + LoRA weight update) against one verifier; VisionClaw doesn't do weight updates (BYO-frontier-API, off-strategy to adopt), but it runs two analogous auto-apply loops that DO optimize fixed verifiers:

- **jury-decides-and-ships** — FIX verdicts auto-queue to an implementer pipeline whose only gate is `tsc`/test pass. The verifier is "verdict label + test gate"; a fix can satisfy both while the proposers actually disagree on *what* to change.
- **AutoTTS knob discovery** — optimizes a proxy-reward over replayed traces (the proxy-reward IS the fixed verifier; see `.agents/memory/autotts-offline-replay.md`).

**Guard implemented for the jury path:** `juryTriage()` now computes **fix-direction concordance** (`fixConcordance` = mean pairwise cosine of the FIX-voting proposers' `fixProposal` embeddings; reuses the embedding infra, no extra frontier calls, best-effort + fail-open to null). The auto-apply seam (`scripts/jury-triage.ts`) refuses to auto-queue a FIX whose proposers agree on the *label* but diverge on the *fix* (`fixConcordance < JURY_FIX_CONCORDANCE_MIN`, default 0.45) — routes to owner-notification instead. This is the held-out/perturbation sanity check: label-agreement is the cheap verifier; fix-direction agreement is the robustness signal the cheap verifier can't see. Fail-open by design (null = single proposal or embeddings down ⇒ never block).

**Guard implemented for the proposal/code-apply path (R125+13.24):** `server/safety/held-out-eval-gate.ts` — a deterministic, LLM-free **held-out eval gate**, the post-apply complement to the pre-apply `fixConcordance` guard. SIA's canonical defense is a data split (agent sees `data/public/`, is graded on never-seen `data/private/`); VC's analog is that the fix-writer optimizes the VISIBLE verifier (in-memory `tsc --noEmit` in `proposal-verifier.ts`) and the cheapest way to pass it is often to ERODE what verifies — delete a test, `.skip` it, strip a sanitizer/tenant filter/`throw`. The gate is a curated invariant registry that the fix-writer's prompt NEVER contains; it runs AFTER tsc passes, over the diff region (oldCode→newCode), and BLOCKS a proposal whose diff nets-out a removal of test blocks/assertions, introduces `.skip`/`.only`, or removes sanitizer calls — so it never reaches `verification_status='passed'` and `safeApplyProposal` won't apply it. Block-severity = test/sanitizer erosion; warn-severity = tenant-ref/throw drop (recorded, non-blocking). Tunable via `HELD_OUT_EVAL_GATE = enforce (default) | warn | off`; fail-OPEN (a throwing gate returns passed). A genuine fix that ADDS code never trips it. Note: the jury NL-queue's "implementer" is the chat agent (no standalone diff-applier script), so enforcement lives at the one deterministic diff gate every materialized proposal passes through — `verifyProposalById`. Tests: `tests/safety/held-out-eval-gate.test.ts`.

**How to apply going forward:** before adding ANY new auto-applied self-improvement lever, ask "what fixed verifier does this optimize, and what's the cheap held-out signal that would catch a fragile fixed point?" Wire that signal as a pre-apply gate. Don't ship a second lever against the same verifier without one. The two halves now exist as a template: `fixConcordance` (pre-apply: do proposers agree on the fix?) + held-out eval gate (post-apply: did the agreed fix quietly weaken a check?).

OpenRouter exposes 200+ models behind one API (Mistral 7B, Llama 3.1 8B, Qwen 2.5 7B, Gemma 2 9B, etc.). VC may be over-paying for proposer diversity by sourcing all-frontier when 8 cheap proposers via OpenRouter could deliver equal-or-better κ-concordance signal at lower spend.
  - **Why this came up:** Nous Hermes Agent's provider matrix (10+ providers, OpenRouter prominent) prompted the gap-check on VC's MoA proposer mix. Single most actionable finding from a Hermes investigation that is otherwise architecturally cousinal-but-shallower than VC.
  - **Proposed R-round shape (not yet scheduled):**
    1. Add OpenRouter as a provider in the model-routing layer (compatible with the existing provider interface — OpenRouter exposes an OpenAI-compatible API).
    2. Add a `proposer_pool: "frontier" | "cheap" | "mixed"` parameter to `ensemble_query` (default `frontier` = current behavior, fail-safe).

  - **Cost ballpark to motivate (rough):** if `ensemble_query` averages 4 frontier calls @ ~$0.02 each = $0.08/call, and 8 cheap @ ~$0.001 each = $0.008/call, that's ~10× cost reduction *if* the quality holds. Even half that savings on the volume of `ensemble_query` calls VC makes per day is meaningful.
  - **Open questions:** (a) does κ-concordance math still discriminate well when proposers are cheap models with narrower stylistic range; (b) does HITL escalation rate go up or down with cheap proposers (could be either — cheaper means more variance which raises κ-disagreement, OR cheaper models converge to similar wrong answers more often which artificially raises κ-agreement); (c) latency profile — 8 parallel cheap calls vs 4 parallel frontier calls — likely cheap wins on wall-clock too, but worth measuring.

## Research nuggets (future-work, no current action)

- **LEANN — graph-based selective recomputation for vector indexes** (Wang et al., MLSys 2026, arXiv:2506.08276; repo `yichuan-w/LEANN`).
  - **Idea:** skip storing dense embeddings; keep a high-degree-pruned proximity graph + raw text only, **recompute embeddings on-demand at query time**. Claims 97% storage savings (60M chunks in 6GB vs 201GB) with no recall loss.
  - **Why we passed (2026-05-16):** trades p95 latency for disk; VC's pgvector+HNSW already hits single-digit-ms recall at our per-tenant volume. C++/MKL/DiskANN build is hostile to Replit Autoscale. No multi-tenant primitives — would need to bolt on `tenant_id` and defeat the design.
  - **When to revisit:** if MNEMA cold-tier (facts queried <1×/month per tenant) ever crosses ~10M vectors per index, the on-demand-recompute trade flips favorable. Pair with a hot/cold tier split: hot stays in pgvector, cold migrates to LEANN-style graph-only storage.
  - **Personal-use note (orthogonal to VC):** LEANN's MCP server is a drop-in semantic-search layer for Claude Code over local files / Apple Mail / browser history / chat exports. Useful as a laptop tool for Bob, not a VC integration.

- **SP-KV — Self-Pruned Key-Value Attention** (arXiv:2605.14037v1, 2026-05).
  - **Idea:** transformer-internal training-time change. A lightweight utility predictor scores each KV pair; only high-utility pairs are *written* to the long-term cache (recent tokens stay in a local window). Jointly trained with the LLM via next-token loss. 3–10× KV-cache shrink → big memory + decode-speed wins with negligible quality loss. Sparsity patterns also serve as an architectural probe for designing hybrid global/local attention layouts.
  - **Why we passed (2026-05-16):** VC is an API consumer, not a model trainer/server. KV cache lives inside provider inference stacks (Anthropic / OpenAI / Gemini / Replit modelfarm / DeepSeek). No knob to inject SP-KV. Even self-hosting an OSS model wouldn't help — SP-KV needs continual pretraining with the utility predictor, not a runtime flag.
  - **When to revisit:** if we ever self-host an open-weights model in-platform (Llama/Qwen/DeepSeek), AND latency/cost on long-context decode becomes the binding constraint. Until then, this benefit arrives for free as providers ship their own KV-cache tricks.
  - **Strategic signal (no action, just awareness):** the field agrees long-context decode is THE cost bottleneck for agentic platforms. Expect long-context token prices to keep falling 6–12 months out. Anything VC has throttled for cost (Felix multi-chapter rollouts, MoA depth, MNEMA k-scoring) likely gets cheaper without code changes.

- **Karpathy's "CLAUDE.md" infographic** (Andrej Karpathy 2026, social-media artifact summarizing his coding-with-agents posture; no formal arXiv).

- **Myopic Planning in LLM Reasoning Traces** (Chen, Li, Cakir, Akcali, Lee, Mattar 2026, NYU/Mattar group, arXiv:2605.06840).
  - **Idea:** interpretability study, not a tool. Built a method to extract literal search trees from LLM chain-of-thought on four-in-a-row, then fit cognitive models to figure out which nodes actually drive the final move. Findings: (1) LLMs expand deep nodes in their CoT but **don't act on them** — move choice is best predicted by a myopic model that ignores deep nodes entirely; (2) performance correlates with search **breadth, not depth**; (3) causal intervention (pruning CoT paragraphs) confirms — removing shallow paragraphs hurts performance, removing deep paragraphs barely matters; (4) this is the *opposite* of human expertise, which is depth-driven. In one line: **the long deep-thinking CoT in reasoning models is largely theater; the real signal is shallow + wide.**
  - **Why this matters for VC (strategic vindication, no integration):** VC has consistently chosen breadth over depth — `ensemble_query` MoA jury (N parallel proposers), chunk-and-parallel orchestration (≤5-min pieces fired in parallel), 8-parallel subagent ceiling, OpenDeepThink action candidate (population-based parallel BT), CoVe R123 (N independent verification questions answered in parallel). This paper validates the architectural bets already made and shifts the burden of proof: "show me the depth helped" rather than "let's assume more thinking = better."
  - **Practical implication:** be miserly with extended-thinking budgets (Anthropic extended-thinking, OpenAI o-series deep reasoning, DeepThink). If move selection is driven by shallow nodes, paying for 32K thinking tokens on a query that a 5× proposer ensemble would solve cheaper is burning money on cosmetic depth. No single platform-wide `thinking_budget` knob to dial down, but audit the surfaces that use extended-thinking and prefer breadth-via-ensemble where the workload allows.
  - **Caveat:** four-in-a-row is a puzzle/strategic-planning domain — the result might be tighter for game-like reasoning than for open-ended generation (writing, code, multi-step tool orchestration). Don't over-generalize to "extended thinking is useless everywhere."
  - **No code, no library, no integration target.** Pure interpretability. Shifts beliefs, not code.

- **GRAM — Generative Recursive Reasoning Models** (Baek, Jo, Kim, Ren, Bengio, Ahn 2026, KAIST + Mila + NYU, arXiv:2605.19376; code "coming soon").
  - **Idea:** transformer-architecture change. Existing recursive reasoning models (HRM, TRM, Looped Transformer) iterate a shared latent transition deterministically — same input → one trajectory → one answer. GRAM makes the transition *stochastic* (learnable Gaussian guidance added to the latent update), so the model samples N parallel reasoning trajectories; a Latent Process Reward Model picks the best one. 97% vs 90.5% on Sudoku-Extreme at comparable compute, solves multi-solution tasks (N-Queens 99.7%, graph coloring) where deterministic recursion mode-collapses.
  - **Why we passed (2026-05-21):** model-internals paper, same fence as SP-KV — VC is an API consumer of frontier providers, no knob to inject GRAM. Code not released yet. Benchmarks are puzzle/constraint-satisfaction tasks, not VC's bottleneck surfaces (longform, tool orchestration, video).
  - **Strategic signal (no action, just awareness):** the *width (N parallel trajectories) + depth (recursion) + reward-model selection* pattern is exactly what VC already does at the agent layer — `ensemble_query` (MoA jury N proposers + κ concordance + HITL when κ<0.5), CoVe verifier (R123 plan→execute→revise with parallel independent verification), chunk-and-parallel orchestration. When frontier providers internalize this natively (Claude/GPT/Gemini ship stochastic recursive reasoning), our agent-layer redundancy gets cheaper because the underlying model already does it. Watch provider release notes for "parallel sampling" / "stochastic reasoning" / "latent process reward" language.
  - **When to revisit:** if VC ever self-hosts an open-weights reasoning model AND GRAM-style fine-tuning becomes a runtime option in HF/vLLM — until then it's free upside on top of provider improvements.

- **turbovec — TurboQuant scalar quantizer for vector search** (Ryan Codrai 2026, github.com/RyanCodrai/turbovec; algorithm: Google Research, arXiv:2504.19874).
  - **Idea:** data-oblivious scalar quantizer (random rotation → Lloyd-Max bucketing → bit-pack). 16× compression at 2-bit (1536-dim: 6,144 B → 384 B). Rust core + Python bindings, hand-written NEON/AVX-512 kernels beat FAISS IndexPQFastScan by 12–20% on ARM, 1–6% at 4-bit on x86. SIMD-level allowlist filtering for hybrid retrieval. No codebook training, no rebuilds, fully local/air-gapped.
  - **Why we passed (2026-05-21):** language mismatch (Rust+Python vs VC's Node/TS — would need a sidecar microservice); pgvector+HNSW already does the job with native SQL tenant/recency/fact-type filtering in one hop; speed win on x86 (Replit prod arch) is marginal (1–6%); RAM-headline ("10M docs in 4 GB") doesn't apply because Postgres pages embeddings to disk — we don't carry the corpus in single-process RAM; solo maintainer, brand-new repo (just hit PyPI/crates.io), maturity risk for a production memory layer.
  - **When to revisit:** (a) if VC ever crosses ~10M embeddings AND needs single-process hot-path retrieval; (b) if we ship a fully on-prem / air-gapped deployment without Postgres; (c) if any subsystem shifts to a Rust/Python stack.

## Gotchas (extracted from replit.md R125+13.13)

- **Drizzle-kit `db:push --force` is destructive** — prefer direct `psql ALTER TABLE`.
- **Google Drive video processing** can take 5–30+ min even on `+faststart` MP4s — use `deliverDigitalProduct()` instant-play routes.
- **`jsdom` with LLM-authored content** — never `runScripts:'dangerously'`.
- **`require()` under ESM** silently fails inside try/catch — use dynamic `await import()`.
- **Transitive CVEs** — `npm audit` weekly (covered by maintenance pass).
- **Tool Sommelier threshold mismatch** — keep `enqueue` and `KEEP` thresholds aligned.
- **OAuth token expiration** — Autoscale envs break in-process schedulers; use the unified auto-refresh system.
- **Cross-tenant leaks** — every storage method MUST enforce `tenantId` in WHERE.
- **Reflexive-loop scale boundary (alphasignalai/Kulkarni 2026)** — auto-revise patterns (Felix grade-then-revise, MoA κ escalation) degrade past ~25k tasks/day in benchmarks: queue stacking + truncated iterations. We have per-job iteration caps but no aggregate-volume circuit breaker. Not an issue at current volume; document the ceiling.
